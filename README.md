## 1. ФОРМУЛИРОВКА ЗАДАНИЯ

**Тема:** Порождающие паттерны GoF: Builder, Prototype, Singleton

**Цель работы:** Изучить порождающие паттерны проектирования «Банды Четырех (GoF)» на примере разработки приложения-конфигуратора компьютерных систем. Освоить на практике паттерны:
- **Строитель (Builder)** — для пошагового конструирования сложных объектов
- **Прототип (Prototype)** — для клонирования объектов с различной глубиной копирования
- **Одиночка (Singleton)** — для обеспечения единственного экземпляра класса с потокобезопасностью

**Задание:** Разработать консольное приложение, моделирующее конфигуратор компьютерных систем, позволяющее:
1. Создавать объекты класса `Computer` (CPU, RAM, GPU, AdditionalComponents)
2. Использовать паттерны Builder и Factory Method для создания конфигураций
3. Использовать Prototype для клонирования (Shallow/Deep Copy)
4. Использовать Singleton для реестра прототипов (PrototypeRegistry)

## 2. ТЕОРЕТИЧЕСКОЕ ОБОСНОВАНИЕ

### Builder (Строитель)

**Назначение:** Пошаговое создание сложных объектов через цепочку вызовов.

**Преимущества:**
- Изолирует код создания от бизнес-логики
- Позволяет создавать разные представления одного объекта
- Упрощает код клиента (цепочка вызовов)

### Prototype (Прототип)

**Назначение:** Создание новых объектов через клонирование существующих.

**Виды копирования:**
| Тип | Описание | Когда использовать |
|-----|----------|-------------------|
| **Shallow Copy** | Копируются поля, но ссылки на объекты остаются теми же | Когда вложенные объекты неизменяемы |
| **Deep Copy** | Создаются новые копии всех вложенных объектов | Когда нужно полное независимое копирование |

### Singleton (Одиночка)

**Назначение:** Гарантия единственного экземпляра класса с глобальной точкой доступа.

**Способы реализации в C#:**
1. **Lazy<T>** — потокобезопасная отложенная инициализация (рекомендуется)
2. **Double-Check Locking** — блокировка с двойной проверкой

**Критика паттерна:**
- Нарушение SRP (смешение бизнес-логики и управления жизненным циклом)
- Глобальное состояние усложняет тестирование
- Скрытые зависимости

## 3. ОПИСАНИЕ ВЫПОЛНЕННЫХ ДЕЙСТВИЙ

### Этап 1: Builder + Factory Method

1. Создан класс `Computer` с полями: CPU, RAM, GPU, AdditionalComponents
2. Создан `ComputerBuilder` с методами:
   - `WithCPU(string)`
   - `WithRAM(int)`
   - `WithGPU(string)`
   - `WithComponent(string)`
   - `Build()`
3. Создан интерфейс `IComputerFactory` с методом `Construct()`
4. Реализованы 3 фабрики:
   - `OfficeComputerFactory` — офисный ПК
   - `GamingComputerFactory` — игровой ПК
   - `HomeComputerFactory` — домашний ПК

**Количество классов:** 7 (Computer, ComputerBuilder, IComputerFactory, 3 фабрики, Program)

### Этап 2: Prototype

1. Добавлены методы в класс `Computer`:
   - `ShallowCopy()` — поверхностное копирование (MemberwiseClone)
   - `DeepCopy()` — глубокое копирование (новый список компонентов)
2. Продемонстрирована разница:
   - Shallow Copy: изменение клона влияет на оригинал (список)
   - Deep Copy: оригинал остаётся неизменным

### Этап 3: Singleton

1. Создан класс `PrototypeRegistry` (Singleton через Lazy<T>)
2. Реестр содержит `Dictionary<string, Computer>` с предопределёнными конфигурациями
3. Метод `GetPrototype(string key)` возвращает **глубокую копию** прототипа
4. Проверено через `ReferenceEquals()` — оба обращения возвращают один экземп
